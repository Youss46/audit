# M19 Clôture d'Exercice — Design Decisions

## Key files
- DB schema: `lib/db/src/schema/closing.ts` — `fiscalYearClosingsTable`
- Closing engine: `artifacts/api-server/src/lib/closing-engine.ts`
- Route: `artifacts/api-server/src/routes/closing.ts`
- Frontend: `artifacts/m15-audit/src/pages/cloture-annuelle.tsx`
- Route registered at: `/cabinet/client/:clientId/cloture`
- Nav tab added to: `ClientAccountingNav.tsx` (5th tab, "Clôture Annuelle")

## Transaction sources added
`"closing_result"` (auto-validated depreciation/finance/net-result entries generated by the closing engine) and `"a_nouveaux"` (opening balance transfer to year+1) were added to `TRANSACTION_SOURCES` in `accounting.ts`.

## Period lock enforcement
- `isPeriodLocked(firmId, clientId, year)` exported from `closing-engine.ts`
- Called in `createTransactionEntry` (accounting.ts) — guards all POST /transactions
- Called in generate-closings handler (fixed-assets.ts) — guards manual dotation generation
- Closing engine itself bypasses the lock by doing direct DB inserts (not via the transaction creation API)

## Net result accounting (SYSCOHADA)
- Class 6 (charges): net debit balance → credit each account to zero them
- Class 7 (produits): net credit balance → debit each account to zero them
- If revenues > expenses: Credit 131 (Résultat net bénéfice)
- If expenses > revenues: Debit 139 (Résultat net perte)
- All closing entries immediately set to `status: "valide"` (no M3 review needed)

## À-nouveaux generation
- Only Classes 1-5 accounts are carried forward (balance sheet accounts)
- The 131/139 result account booked in step 2 IS included (it's in Class 1)
- Net balance per account: positive = debit balance in opening entry, negative = credit
- Single transaction dated Jan 1 of year+1, source = "a_nouveaux"

## Role restriction
- `POST /ledger/close-period` restricted to `expert_comptable` only
- Frontend hides/disables the final button for non-expert_comptable roles

## DB unique constraint
`unique("fiscal_year_closings_firm_client_year_unique")` on (firmId, clientId, year) — prevents double-closing; `onConflictDoUpdate` in the engine updates if somehow called twice (though isPeriodLocked guard prevents it at application level).

**Why:** The closing is a one-way, permanent operation. The unique constraint + application-level lock check provide defense in depth.
